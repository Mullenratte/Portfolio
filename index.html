About

Golem Game is set to be an Action-RPG that features mechanics inspired by Noita and Monster Hunter. In its core loop, you explore the world, fight against various monsters and collect their resources to craft your own unique spells.

Introduction

I started this Unreal Engine 5.5 project in April 2025 and have since been working on it almost every day. My main focus in the beginning was creating the spell system as the foundation of gameplay. This was done mainly in C++.
I then spent some time learning about Unreal's Gameplay Ability System and used it to implement some basic spells and a Dodge ability for the player.
Most recently, I started working on the enemy AI. I setup a basic behavior tree that allows the enemy to follow a path of spline points and recognize the player if they get within a certain sight radius. The enemy will then chase the player and, when in range, opt for one of multiple possible gameplay abilities. The chosen ability is determined by an evaluation of the current situation based on different heuristics, e.g. the distance from the target.

Spell System

I wanted the player to be able to create their own spells and get really creative in the process. The system features Spells and Modifiers and was heavily inspired by Noita. Spells can be cast on their own, but can also be combined with modifiers and other spells to create new, more powerful spells.
A modifier usually only applies to the next entry in the component list. Putting the Duplicate Modifier before an Arcane Missile for example, results in a Spell that casts 2 Arcane Missiles. Doing the same, but adding another Arcane Missile at the end like this results in a spell that casts 3 Arcane Missiles, because the Duplicate Modifier only applies to the first Arcane Missile.

Now, what happens when we do this: ? We get 4 Arcane Missiles.
So this leads to a spell that casts 8 Arcane Missiles!

Utilizing the Duplicate Modifier to create an octuple spell.


We can see that these modifiers seem to stack, but how does it work internally?
Any spell the player can cast is a UCastableSpell. When combining spells and modifiers in the spell crafting menu, the system internally calls the static CreateCastableSpell function of the UCastableSpellFactory class. This basically takes a TArray of the components as parameter. It then iterates through the array in reverse order and saves all modifiers in a TArray.

Note: This is important, since modifiers always affect subsequent entries. These entries can not be Modifiers themselves, which is why they need to be applied in reverse order.

The modifiers are then applied. Generally speaking, modifiers always do the following:

  1. Create a new empty USpellComposition
  2. Initialize the spell composition with all spells (or spell compositions!) the modifier affects
  3. Apply the actual modifier effect to the spell composition
  4. Update the Factory's Components list by replacing itself and all modified components with the new Spell Composition

Example:
We have an Arcane Missile and a Fireball spell. The Chain Modifier chains these two spells together into one spell that casts Fireball and Arcane Missile at roughly the same time.

The spell components' corresponding Unreal C++ base classes.

This Castable Spell can then be assigned to one of the player character's wand slots and is ready to be used in combat!

Improvements & Conclusion

The system is by no means implemented in a perfectly clean manner. For example, the system currently expects each Modifier class to directly access and modify the factory's components list. I am aware this is prone to error and potentially tedious to debug. Having the Apply function return the spell composition and the indices to remove as out parameters would be one way to achieve a cleaner solution. Also, since all Modifiers share a lot of the same logic, it's worth to think about introducing a common base class instead of an interface.
As for right now, the system is working as intended and suffices for the current stage the game is in. I will definitely give it an overhaul with the mentioned improvements though, once I find the time.
Nonetheless, although the system is still a bit rough around the edges, I am very pleased with the outcome and the things I learned during the implementation. It's the foundation for a potentially extremely enjoyable Magic System that enables creative and exploratory urges in players.

Thank you for reading through all of this, I hope this was insightful!
